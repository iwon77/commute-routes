# 2026-06-20 Tech Stack Rationale

## 개요

- 이 문서는 현재 MVP 기준 기술 스택과 핵심 선택 이유를 기록한다.
- 모든 논의를 남기지 않고, 이후 설계나 구현 판단에 직접 영향을 주는 선택만 정리한다.

## 현재 기술 스택

- Frontend: `React`, `TypeScript`, `Vite`
- Client Storage: `IndexedDB`
- Backend: `Java`, `Spring Boot`, `Spring Web`
- Database: `PostgreSQL`
- Database Access: `JdbcTemplate`
- 확장 후보: `MyBatis`

## 브라우저 저장소 선택

### 결정

- 사용자 경로 데이터 저장소는 `IndexedDB`를 사용한다.

### 비교 대상

- `localStorage`
- `IndexedDB`

### `localStorage`를 선택하지 않은 이유

- 문자열 기반 저장이라 목적지, 경로 옵션, 단계 같은 구조화 데이터를 다루기 불편하다.
- 데이터가 늘어나면 직렬화와 역직렬화 부담이 커진다.
- 동기 API라 저장/조회 시 UI에 부담을 줄 수 있다.
- 목록, 상세, 수정, 삭제가 있는 다단계 데이터 모델을 안정적으로 다루기에 한계가 있다.

### `IndexedDB`를 선택한 이유

- 구조화된 데이터를 브라우저 안에 저장하기에 더 적합하다.
- 비동기 방식이라 UI 흐름과 분리하기 쉽다.
- 목적지, 경로 옵션, 단계 같은 다건 데이터 관리에 맞는다.
- 새로고침 이후에도 사용자 경로를 유지해야 하는 현재 MVP 요구사항과 잘 맞는다.
- 이후 검색 결과 일부나 화면 보조 데이터를 확장 저장해야 할 때도 대응하기 쉽다.

### 이 선택이 현재 프로젝트에 맞는 이유

- 이 프로젝트는 사용자 경로를 서버 DB가 아니라 브라우저에 저장한다.
- 경로 구조가 단순 설정값이 아니라 `목적지 -> 경로 옵션 -> 단계`의 다단계 구조다.
- 따라서 `localStorage`보다 `IndexedDB`가 현재 요구사항에 더 직접적으로 맞다.

## 서버 DB 접근 방식 선택

### 결정

- 서버 DB 접근은 `JdbcTemplate`로 시작한다.
- 조회 복잡도가 커질 때만 `MyBatis` 확장을 검토한다.

### 비교 대상

- `JdbcTemplate`
- `MyBatis`

### `MyBatis`를 바로 선택하지 않은 이유

- 현재 MVP 단계에서는 서버 역할이 사용자 경로 저장보다 메타데이터 조회, 실시간 도착정보 조회, 캐시 처리에 집중된다.
- 초기에는 쿼리 수와 동적 조건이 아주 많다고 보기 어렵다.
- 지금은 빠르게 시작하고 SQL을 직접 통제하는 쪽이 구현 속도와 가시성 면에서 유리하다.
- `MyBatis`는 조회가 복잡해질수록 장점이 커지지만, 현재 시점에는 초기 설정 비용이 상대적으로 더 크다.

### `JdbcTemplate`를 선택한 이유

- Spring 기본 구성 안에서 가볍게 시작할 수 있다.
- SQL이 직접 드러나므로 동작을 예측하고 디버깅하기 쉽다.
- 메타데이터 조회, 캐시 조회/저장 같은 현재 서버 책임과 잘 맞는다.
- JPA처럼 객체 연관관계를 적극 활용하는 설계가 현재 프로젝트와 맞지 않는다.
- MVP 초기에는 추상화보다 명시적인 쿼리 제어가 더 중요하다.

### 이후 `MyBatis`를 검토할 조건

- 검색 조건 조합이 많아질 때
- 기관별/지역별/교통수단별 분기가 늘어날 때
- 동적 SQL 관리 비용이 `JdbcTemplate` 코드에서 커질 때
- 매퍼 단위로 SQL을 분리 관리하는 편이 더 유리해질 때

## JPA를 우선하지 않은 이유

- 현재는 `Long` 타입 ID 중심 저장을 전제로 하고 있다.
- 객체 연관관계를 적극적으로 매핑하는 방향이 아니다.
- 사용자 경로 데이터도 서버 DB가 아니라 브라우저에 저장한다.
- 따라서 현재 서버는 도메인 객체 그래프를 다루는 구조보다 조회/정규화/캐시 중심 구조에 가깝다.
- 이 상황에서는 JPA의 장점보다 SQL 직접 제어의 장점이 더 크다.

## 프론트엔드 조합 선택 배경

### 결정

- 프론트는 `React + TypeScript + Vite` 조합을 사용한다.

### 이유

- 다단계 경로 편집 UI와 로컬 상태 관리가 필요한 구조에 잘 맞는다.
- TypeScript로 단계 타입, 식별자, 조회 응답 구조를 명확하게 다룰 수 있다.
- Vite는 개인용 MVP 초기 부트스트랩과 개발 속도에 유리하다.
- 서버는 Spring으로 가져가되, 클라이언트의 IndexedDB 활용과 편집 UI는 SPA 구성이 더 자연스럽다.

## 데이터베이스 선택 배경

### 결정

- 서버 DB는 `PostgreSQL`을 사용한다.

### 이유

- Spring 생태계와 연결이 안정적이다.
- 메타데이터와 캐시 저장용으로 무난하고 확장성도 충분하다.
- MVP 이후 검색 조건이나 운영 데이터가 늘어나도 대응이 쉽다.

## 정리

- 사용자 경로 저장은 구조화된 브라우저 저장이 필요해서 `IndexedDB`를 선택했다.
- 서버 DB 접근은 현재 역할과 MVP 속도를 고려해 `JdbcTemplate`로 시작한다.
- `MyBatis`는 서버 조회 복잡도가 실제로 커졌을 때 확장 후보로 둔다.
- 현재 기술 스택 선택은 “개인용 MVP를 빠르게 구현하되, 이후 확장 여지는 남기는 방향”에 맞춰졌다.
