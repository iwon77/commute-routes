# commute-routes AGENTS.md

## 프로젝트 목적과 저장소 역할

- `commute-routes`는 출퇴근 경로와 단계별 이동 정보를 다루는 프로젝트다.
- 현재 저장소는 프로젝트 판단 근거, 기술 결정, 구현 준비 문서를 관리하는 기준 저장소로 사용한다.
- 사용자 경로 자체는 서버 DB가 아니라 브라우저 저장소에 저장한다.
- 브라우저 저장소 기본 선택은 `IndexedDB`다.
- 서버 `PostgreSQL`은 정류장, 역, 노선 메타데이터와 외부 API 캐시 용도로 사용한다.
- 서버 DB 접근은 우선 `JdbcTemplate` 기준으로 설계하고, 조회 복잡도가 커질 때만 `MyBatis` 확장을 검토한다.

## 코딩 판단 원칙

- 먼저 현재 요구사항이 문서 정리인지, 설계 결정인지, 구현 요청인지 구분한다.
- 구현 전에 가정, 대안, 제외 범위를 짧게 드러낸다.
- 아직 확정되지 않은 기능을 미리 일반화하지 않는다.
- MVP 범위를 벗어나는 기능은 요청이 없으면 넣지 않는다.
- 사용자 경로 저장, 단계별 도착 정보, 환승 관련 로직은 현재 문서화된 기준을 먼저 따른다.
- 직접 관련 없는 리팩터링, 포맷 정리, 구조 변경은 요청 없이는 하지 않는다.

## 구현 및 검증 원칙

- 작업마다 성공 기준을 먼저 다시 적는다.
- 버그 수정이면 재현 조건이나 실패 시나리오를 먼저 정리한다.
- 기능 추가면 기대 동작과 확인 포인트를 먼저 정리한다.
- 변경 후에는 가능한 범위에서 테스트, 빌드, 수동 확인 중 맞는 방법으로 검증한다.
- 테스트를 수행하지 못했으면 이유를 명시한다.
- 단계별 도착 정보 로직을 다룰 때는 `effectiveTime = now + transferMinutes` 기준이 유지되는지 확인한다.

## 작업 기본 워크플로우

- 작업 시작 시 `검토 내용`, `결정 필요 사항`, `영향 범위`, `리스크`, `추천안`을 먼저 정리한다.
- 사용자가 구현 진행을 확인하면 그때 코드나 문서를 수정한다.
- 구현 범위는 짧고 명확하게 다시 적고, 필요한 파일만 수정한다.
- 검증 결과를 남기고, 사용자가 이어서 확인할 수 있게 변경 요지를 짧게 설명한다.
- 마무리 단계에서 필요하면 커밋 메시지와 PR 문구를 작성한다.

## 작업 시작 시 응답 원칙

- 바로 수정하지 말고 먼저 무엇을 바꾸는지와 왜 그렇게 바꾸는지 정리한다.
- 해석이 둘 이상 가능하면 차이를 먼저 적고, 임의로 하나를 고르지 않는다.
- 핵심 동작에 영향을 주는 모호함이 있으면 구현 전에 먼저 확인한다.
- 구현 후에는 사용자가 직접 검증할 수 있는 포인트를 짧게 전달한다.

## 현재 기능 기준 메모

- 다음 단계 도착 정보는 현재 시각 즉시 도착편이 아니라, 이전 단계 이동시간 이후 도착편만 표시한다.
- 기준 시간은 `effectiveTime = now + transferMinutes`다.
- `transferMinutes`는 사용자 입력값이 있으면 우선하고, 없으면 평균 도보 이동시간 또는 이후 도입할 길찾기 API 계산값을 사용한다.
- 환승 가능 여부 판정 기능은 현재 MVP 범위에서 제외한다.

## 브랜치 전략

- 기본 브랜치는 `main`이다.
- 별도 규칙이 없으면 작업 브랜치는 `feat/*`, `fix/*`, `docs/*` 중 목적에 맞는 접두사를 사용한다.
- 이슈 제목, 브랜치명, 작업 범위가 있으면 그 기준을 우선한다.

## 문서 구조

- 루트 문서는 프로젝트 기준 규칙과 저장소 공통 문서를 둔다.
- `archive/`는 프로젝트 진행 로그와 판단 근거 문서를 둔다.
- 구현 코드, 설계 문서, 외부 연동 문서가 생기면 그 구조를 이 파일에 추가로 정의한다.

## 외부 저장소 및 참조 규칙

- 현재 확인된 외부 저장소 전용 규칙은 없다.
- 외부 저장소나 별도 서비스 문서를 참조해야 하면, 이 저장소 기준과 충돌하는지 먼저 확인한다.
- 충돌 시 현재 저장소의 확정 문서와 코드 기준을 우선하고, 필요한 차이는 `archive/`에 기록한다.

## archive/ 사용 기준

- `archive/`는 프로젝트 진행 로그 저장소다.
- 아래 종류의 문서를 `archive/`에 저장한다.
- 기술 결정과 판단 근거
- 요구사항 해석과 범위 정리
- 구현 방향 비교와 보류 사유
- 다음 작업자가 이어서 볼 수 있는 작업 맥락과 메모
- 개인 회고, 일반 학습 노트, 전체 세션 대화 원문은 프로젝트 `archive/`에 저장하지 않는다.
- 파일명은 가능하면 날짜와 주제를 함께 드러내는 형식을 사용한다.
